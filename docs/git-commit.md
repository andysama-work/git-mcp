## Git 提交规范

### 提交前流程

**重要：每次提交前必须遵循以下流程**

1. **生成变更导图**
   - 展示文件变更、新增功能、修复问题等

2. **确认用户**
   - 等待用户确认变更内容
   - 确认无误后再进行提交

3. **执行提交**
   - 仅执行本地提交（不自动 push）
   - 使用标准的提交信息格式


### 提交信息格式

#### 格式结构

```
<emoji> <type>: <简短描述>

详细描述：
- 变更点 1
- 变更点 2
- 变更点 3
```

#### Type 类型和对应表情

| Type     | Emoji | 说明                                   |
| -------- | ----- | -------------------------------------- |
| feat     | ✨     | 新增功能                               |
| fix      | 🐛     | 修复 Bug                               |
| docs     | 📝     | 文档变更                               |
| style    | 💄     | 代码格式（不影响代码运行的变动）       |
| refactor | ♻️     | 重构（既不是新增功能，也不是修复 Bug） |
| perf     | ⚡️     | 性能优化                               |
| test     | ✅     | 增加测试                               |
| chore    | 🔧     | 构建过程或辅助工具的变动               |
| build    | 📦     | 构建系统或外部依赖变动                 |
| ci       | 👷     | CI 配置文件和脚本变动                  |
| revert   | ⏪     | 回退代码                               |
| init     | 🎉     | 项目初始化                             |
| ui       | 🎨     | 更新 UI 和样式文件                     |
| config   | ⚙️     | 配置文件修改                           |
| merge    | 🔀     | 合并分支                               |

#### 提交信息示例

##### 示例 1: 新增功能

```
✨ feat: 实现用户登录功能

详细描述：
- 添加登录表单组件
- 实现 JWT token 认证
- 添加登录状态管理
- 集成 API 接口调用
- 添加表单验证逻辑
```

##### 示例 2: 修复 Bug

```
🐛 fix: 修复用户头像上传失败问题

详细描述：
- 修复文件大小限制验证逻辑
- 添加图片格式检查
- 优化错误提示信息
- 更新上传接口参数
```

##### 示例 3: 代码重构

```
♻️ refactor: 重构用户认证模块

详细描述：
- 提取公共认证逻辑到 Hooks
- 优化组件结构
- 移除冗余代码
- 改进类型定义
- 统一错误处理方式
```

##### 示例 4: 性能优化

```
⚡️ perf: 优化列表渲染性能

详细描述：
- 使用 React.memo 避免不必要的重渲染
- 实现虚拟列表减少 DOM 节点
- 添加数据懒加载
- 优化图片加载策略
```

##### 示例 5: 配置变更

```
⚙️ config: 配置 Tailwind CSS 和代码格式化

详细描述：
- 安装并配置 Tailwind CSS
- 添加 Prettier 代码格式化
- 配置 ESLint 规则
- 设置路径别名
- 创建 VSCode 工作区配置
```

##### 示例 6: 文档更新

```
📝 docs: 更新项目 README 文档

详细描述：
- 添加项目特性说明
- 更新安装步骤
- 添加开发命令说明
- 完善路径别名文档
- 添加代码格式化指南
```

### 提交注意事项

1. **简短描述**
   - 不超过 50 个字符
   - 使用动词开头
   - 不使用句号结尾
   - 清晰描述做了什么

2. **详细描述**
   - 与简短描述空一行
   - 使用列表形式
   - 说明为什么做这个变更
   - 说明具体做了哪些改动
   - 每个变更点独立一行

3. **提交粒度**
   - 一次提交只做一件事
   - 相关的改动放在一起提交
   - 不要混合不同类型的变更

4. **提交频率**
   - 完成一个功能点就提交
   - 修复一个 Bug 就提交
   - 不要累积大量变更后一次性提交

### Git 工作流程

```bash
# 1. 查看当前变更
git status

# 2. 添加文件到暂存区
git add .

# 4. 确认变更（等待用户确认）
# 5. 执行提交（使用规范的提交信息）

git commit -m "✨ feat: 简短描述

详细描述：
- 变更点 1
- 变更点 2
- 变更点 3"

# 6. 推送到远程（需要用户手动执行）
git push origin main
```

### 禁止的操作

❌ 不允许的提交信息：
```
git commit -m "update"
git commit -m "fix bug"
git commit -m "修改"
```

❌ 不允许直接 force push：
```
git push --force
```

❌ 不允许提交未格式化的代码

✅ 提交前检查清单：
- [ ] 代码已格式化
- [ ] 通过 ESLint 检查
- [ ] 通过 TypeScript 类型检查
- [ ] 生成并确认了变更导图
- [ ] 提交信息符合规范
- [ ] 只提交相关变更

Always generate Git commit messages in Chinese.
Git 提交信息必须使用中文生成。

### 输出格式要求

**重要：生成提交信息时，必须使用 markdown 代码块（三个反引号）包裹提交信息，方便用户复制。**

正确示例：
```
🔧 chore: 升级版本至1.0.11

详细描述：
- 将应用版本从1.0.10升级到1.0.11
```

错误示例（不要这样做）：
直接输出纯文本，不用代码块包裹。
